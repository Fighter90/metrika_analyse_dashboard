# ADR-007 — Парсер Метрики и хранение истории по дням (SQLite)

- **Статус:** принято
- **Дата:** 2026-05-22
- **Контекст:** инструмент должен перед анализом выкачивать максимум данных из Метрики и
  хранить историю по периодам/дням. Рассматривался вариант с БД в Docker.

## Решение

- **Хранилище — локальная SQLite** (`better-sqlite3`), без Docker. Подтверждает ADR-002.
- **Парсер (ETL) `pnpm sync`** выкачивает агрегаты Stat API (§5.4) дневными чанками по всем
  целям (с порогом архивации), сохраняя **сырьё** в `raw_responses` и нормализуя в
  `channel_stats` и др. Дашборд/отчёты читают только БД, не живой API.
- **История по дням:**
  - `raw_responses(query_hash, date_from, date_to)` — UNIQUE, идемпотентный кэш по периоду.
  - `channel_stats(date, channel, utm_*)` — PRIMARY KEY включает `date` → данные копятся по дням.
  - Повторный `sync` дополняет историю, не затирая прошлые дни (`INSERT … ON CONFLICT`).
- **Глубина:** агрегаты Stat API. Logs API (сырые хиты) — вне scope; включение потребует
  отдельного ADR и, вероятно, пересмотра хранилища (объём).

## Обоснование

- Цель спеки: `./run.sh` за <2 мин у волонтёра без backend-опыта. Docker/Postgres добавляют
  обязательный Docker Desktop и время старта — против этой цели.
- SQLite уже даёт историю по дням, offline и воспроизводимость отчётов (immutable snapshot).
- Объём агрегатов Stat API для одного счётчика умещается в SQLite без проблем.

## Последствия

- Для WoW/трендов достаточно запросов по диапазону `date`.
- Если в будущем понадобится Logs API или многопользовательский доступ — отдельный ADR
  с переходом на Postgres (возможно в Docker).

## Альтернативы

- Postgres в Docker — отклонено сейчас: лишняя инфраструктура для локального инструмента.
