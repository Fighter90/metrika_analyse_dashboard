# ADR-008 — Server-side рендер графиков в отчёт (DOCX/PDF)

- **Статус:** предложено (ожидает решения по зависимости)
- **Дата:** 2026-05-27
- **Контекст:** fix-prompt FINAL §6.4 требует встраивать графики **изображениями** в DOCX и PDF
  так, чтобы из одного `snapshotId` DOCX и PDF давали **идентичный контент** (SHA-256 без учёта
  zip/pdf-таймстемпов). Сейчас отчёт — текст + таблицы (детерминированный, без LLM/часов в
  render-пути); интерактивные графики есть только на дашборде (ECharts в браузере).

## Проблема

- **PDF** рендерится через Puppeteer (HTML → PDF) — туда можно встроить inline-SVG/`<img>`.
- **DOCX** (`docx` npm) требует **растровое** изображение (PNG/JPEG) в `ImageRun`; SVG не
  поддерживается переносимо между версиями Word.
- Значит, нужен способ получить **PNG графика на сервере**. ECharts SSR умеет только SVG.
  Конвертация SVG→PNG в Node требует нативной зависимости (`sharp` / `@resvg/resvg-js` / `canvas`),
  а это — **вне зафиксированного стека** (CLAUDE.md: новые зависимости только через ADR).
- Инвариант «DOCX==PDF по контенту» делает фичу **all-or-nothing**: нельзя поставить графики
  только в PDF — иначе SHA разойдётся и нарушится детерминизм.

## Рассмотренные варианты

| Вариант                                                                                      | Новая зависимость                                                                | Плюсы                                                                                                                             | Минусы                                                  |
| -------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------- |
| **A. Puppeteer + ECharts** — рендерить опцию графика в headless-странице, `screenshot` → PNG | нет рантайм-зависимости (Puppeteer уже есть для PDF); ECharts уже в стеке фронта | браузер в build-пути снапшота (медленно); шрифто-зависимый детерминизм между машинами; код браузера вне coverage (как `buildPdf`) |
| **B. `@resvg/resvg-js`** — ECharts SSR (SVG) → resvg → PNG                                   | да (нативная)                                                                    | быстро, без браузера, стабильный растр                                                                                            | новая нативная зависимость; сборка под разные платформы |
| **C. `sharp`** — SVG → PNG                                                                   | да (нативная)                                                                    | зрелая                                                                                                                            | тяжёлая нативная зависимость                            |

## Рекомендация

**Вариант A** (Puppeteer + ECharts): не добавляет новой рантайм-зависимости (переиспользуем уже
существующий Puppeteer и ECharts из фронта). PNG-байты рендерятся **один раз** при сборке снапшота
и сохраняются в снапшот (`snapshot.charts: { id: base64 }`), а render-путь DOCX/PDF лишь читает
сохранённые байты — детерминизм соблюдён (как с `aiNarrative` / `generatedHypotheses`).

Детерминизм между машинами (одинаковый PNG → одинаковый SHA) обеспечивается фиксацией версии
Chromium (Playwright уже пинит билд), отключением анимаций ECharts (`animation:false`) и фиксированным
`devicePixelRatio`/размером. Если кросс-машинный SHA окажется нестабильным, инвариант смягчается до
«DOCX и PDF строятся из одних и тех же сохранённых PNG» (контент идентичен, побайтовый PDF/zip — нет).

## Последствия

- Новый модуль `code/backend/src/report/chart-renderer.ts` (Puppeteer) — **вне coverage**
  (как `pdf/renderer.ts`), покрывается e2e.
- Сборка снапшота становится медленнее (запуск браузера) — допустимо для локального инструмента.
- Реализация выделена в отдельную итерацию (v2.7.1): рендер 3 ключевых графиков (бар каналов,
  воронка, микс каналов) + встраивание в `buildDocx`/`reportHtml` + e2e на наличие изображений.

## Что НЕ делаем

- Не добавляем `sharp`/`resvg`/`canvas` без отдельного одобрения (варианты B/C остаются запасными,
  если вариант A окажется ненадёжным по детерминизму).
