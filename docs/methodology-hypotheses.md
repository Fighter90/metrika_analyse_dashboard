# Методология гипотез (проблемы → решения)

> 🇷🇺 Русский. Методология формулирования и проверки продуктовых гипотез для трека
> «Конверсии и лидген» ProductCamp. Гипотезы **генерируются автоматически из данных Метрики**
> (локальная SQLite), а не вводятся вручную. Цикл — Double Diamond (Define → Develop → Deliver).
> Любое изменение методологии фиксируется здесь с обоснованием.

## 1. Откуда берутся гипотезы

Гипотезы формирует **AI-анализ** на основе данных, спарсенных из Яндекс.Метрики и сохранённых в
локальной БД (каналы, UTM, гео/устройства, страницы входа/выхода, воронка, B2B). Команда их **не
вводит руками** — система предлагает их после анализа; человек принимает/уточняет и фиксирует исход
в Decision Log. Каждая цифра прослеживается до `raw_responses` (anti-hallucination).

## 2. Гипотеза проблемы (problem hypothesis)

**Формула:**

> **[Типичный пользователь]** испытывает **[конкретную трудность]** при **[действии/цели]**,
> потому что **[что-то мешает]**.

- **Типичный пользователь** — конкретный сегмент (напр. «посетитель с мобильного из России»).
- **Конкретная трудность** — боль/неудовлетворённость.
- **Действие/цель** — сценарий, где трудность возникает (напр. «при заполнении формы регистрации»).
- **Что-то мешает** — конкретный барьер в продукте.

**Объём:** минимум **10 гипотез проблем**, выведенных из данных (слабые сегменты, низкий CR,
отвалы на страницах, неконвертящие каналы).

## 3. Гипотеза решения (solution hypothesis)

**Формула:**

> Если **[мы сделаем ___]**, то **[пользователи смогут ___]**, что приведёт к
> **[результату для бизнеса]**.

- **Мы сделаем** — конкретное действие (функция, изменение, сервис).
- **Пользователи смогут** — какую проблему/потребность решаем.
- **Результат для бизнеса** — какие метрики улучшим (конверсия, удержание, выручка, снижение затрат).

**Объём:** минимум **10 гипотез решений**, каждая привязана к конкретной проблеме.

### Чек-лист хорошей гипотезы решения

- [ ] Содержит предположение о влиянии на пользователя (что изменится в UX).
- [ ] Есть явная связь с проблемой пользователя.
- [ ] Формулировка конкретна, а не общая («снизим время ожидания с 2 мин до 30 сек», не «улучшим сервис»).
- [ ] Есть ожидаемый эффект для бизнеса (влияние на метрику).
- [ ] Проверяема: можно измерить «подтвердилась/опроверглась».
- [ ] Содержит критерий успеха («увеличим клики на 10%», «снизим отток на 5%»).
- [ ] Сфокусирована: одна идея = одна гипотеза.

### Как ищем решения

Бенчмаркинг/конкурентный анализ, брейншторминг (кросс-функционально), customer co-design. Если
проблема конкретна — решение часто очевидно, не закапываемся.

## 4. Риски решения

Для каждой solution-гипотезы оцениваются риски:

- **Ценность (Value risk)** — решение может не дать ценности пользователю.
- **Удобство (Usability risk)** — пользователи не смогут им воспользоваться.
- **Технические (Feasibility risk)** — сложно/дорого реализовать и поддерживать.
- **Бизнес (Business viability)** — не окупится или вредит бизнес-модели.
- **Юридические / репутационные.**

## 5. Приоритизация

ICE = **Impact × Confidence × Ease** (произведение, шкала 1–1000, см.
`docs/decisions/005-ice-product-vs-mean.md`) **с письменным обоснованием каждого балла**.

> ⚠️ Open question (`docs/specs/002-*`): во вводных были и «не приоритизировать по ICE», и
> «приоритизировать по ICE с обоснованием». По умолчанию — **ICE с обоснованием**; финальное решение
> фиксируется в спеке/ADR.

## 6. План проверки решения

Для каждой приоритетной гипотезы решения:

1. **Что проверяем:** ценность, удобство, бизнес-эффект, влияние на контр-метрики.
2. **Методы:** интервью/опрос · прототип/клик-тест · Fake-Door / лендинг · A/B-тест.
3. **Эксперимент:** целевая аудитория, канал привлечения, материалы, критерии успеха («≥X% …»).
4. **Результаты:** факт vs критерии, инсайты, решение (отбросить / скорректировать / в MVP).

## 7. Decision Log

Решение по каждой проверенной гипотезе фиксируется записью DL-{N}: что проверяли, что узнали,
доказательства, исход (🟢/🟡/🔴), следующий шаг. Раздел Decisions заполняется **после** AI-анализа.
