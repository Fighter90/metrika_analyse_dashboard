# 002 — Переработка продукта: AI-гипотезы, отчёт по ГОСТ, дашборд, ops

> Статус: `accepted` · Автор: Sergey Emelyanov · Дата: 2026-05-22 · PR: (по фазам)
>
> Заменяет [001-dual-goal-kpi](001-dual-goal-kpi.md) (`superseded`): KPI-разделение «заявка ≠ оплата»
> сохраняется в анализе, но ручной редактор гипотез убирается — гипотезы генерирует AI.

## 1. Проблема и контекст

Текущий продукт требует **ручного** ввода гипотез и решений, отчёт короткий, дашборд «скудный»
(мало графиков/фильтров), нет управления данными из UI. Команде трека (саммари встречи —
`docs/productcamp_conversion_summary_19_05.md`) нужен **аналитический инструмент**, который сам из
данных Метрики формирует подробный отчёт с гипотезами под цель «300 платных билетов».

## 2. Цель

Превратить инструмент в **AI-аналитика**: по данным локальной БД (спарсенным из Метрики) система
сама формулирует гипотезы проблем и решений, строит детальный отчёт (≥40 стр. A4, ГОСТ) с графиками,
а дашборд даёт богатую визуализацию с фильтрами по датам. Управление данными (ключ/счётчик/обновление)
— из UI.

**Принципы (неизменны):** anti-hallucination (каждое число → `raw_responses`); «заявка ≠ оплата»;
детерминизм отчёта (нет `Date.now()`/LLM в render-пути — AI-контент генерится один раз и сохраняется
в снапшот); 100% покрытие тестами.

## 3. Подтверждённые решения

- **Приоритизация — ICE × обоснование** (Impact×Confidence×Ease, произведение, см. ADR-005).
- **Settings в UI** — ввод OAuth-токена/ключа/счётчика, «Очистить данные»/«Сохранить»/«Заполнить»
  (re-parse). Секреты хранятся локально (.env/БД, gitignored), в репозиторий не попадают.
- **Методология гипотез** — `docs/methodology-hypotheses.md` (новые формулы problem/solution).

## 4. Фазы (каждая — отдельный PR с тестами и 100% покрытием)

### Фаза A — Движок гипотез (ядро)

- AI генерирует **≥10 проблемных + ≥10 решенческих** гипотез из данных снапшота (каналы/UTM/гео/
  страницы/воронка/B2B), по формулам из `methodology-hypotheses.md`.
- Проблема: «[пользователь] испытывает [трудность] при [действии], потому что [барьер]».
- Решение: «Если [сделаем], то [пользователи смогут], что приведёт к [результату]» + чек-лист качества.
- Каждой solution-гипотезе — **риски** (Value/Usability/Feasibility/Business/Legal) и **план проверки**
  (что проверяем, методы, эксперимент, критерии успеха).
- Приоритизация по **ICE с обоснованием** каждого балла.
- Разделы **Hypotheses/Decisions заполняются автоматически после AI-анализа**, ручной редактор убран
  (или переведён в режим просмотра/уточнения). Decisions — производные от проверки гипотез.
- AC: ≥10+≥10 гипотез из реальных данных; каждая привязана к цифре из `raw_responses`; детерминизм
  (один снапшот → один набор, AI-вызов один раз, результат сохранён).

### Фаза B — Отчёт ≥40 стр. A4 по ГОСТ

- Структура: титул (ГОСТ), оглавление, **аналитика 10–15 стр.** (каналы/UTM/гео/устройства/страницы/
  воронка/динамика, выводы под цель встречи), **гипотезы 10–15 стр.** (проблемы → решения → риски →
  план проверки → приоритизация ICE), Decision Log, приложение данных. Итого **30–40 стр.**
- Графики в отчёте (из данных снапшота).
- **DOCX и PDF** содержат AI-анализ + гипотезы; кнопки **Export DOCX / Export PDF реально скачивают**
  файлы на компьютер; кнопка **«Перестроить отчёт»**. На экране — читаемый форматированный отчёт.
- AC: один `snapshotId` → байт-идентичные DOCX/PDF; ≥40 стр. на реальных/демо-данных; оформление ГОСТ.

### Фаза C — Дашборд и графики

- Больше графиков (особенно на **Overview**); **фильтры по датам**, распределение визитов/заявок по
  датам; больше **легенд**; подробные тултипы; «как в Метрике».
- **Только релевантные отчёту графики** — на дашборд выводим именно те визуализации, на которых
  строятся анализ и гипотезы (а не всё подряд из Метрики). Дашборд и отчёт используют один набор
  данных/срезов.
- AC: каждая страница — фильтр периода влияет на графики; легенды/единицы/форматирование; пустые
  состояния; данные совпадают с API и `raw_responses`.

### Фаза D — Ops UI

- Большая кнопка **«Обновить данные из Яндекс.Метрики»** (re-parse → локальная БД → перегенерация
  отчётов).
- **«История отчётов»** — список снапшотов по датам, история запросов/генераций, просмотр прошлого.
- **«Настройки»** — OAuth-токен/ключ/счётчик, «Очистить все данные»/«Сохранить»/«Заполнить данные»
  (смена счётчика → переналив).
- Удалить/переосмыслить раздел **«Sources / Откуда эта цифра»** (прослеживаемость остаётся как
  свойство данных, отдельная страница убирается).

### Фаза E — Корректность, тесты, документация

- **Аудит парсинга Метрики**: проверить, что выгрузка корректна (атрибуты, проценты→доли, дневные
  чанки, totals-формы, best-effort разбивки).
- Папка **Tests** (или существующая пирамида) — **unit / функциональные / приёмочные** на весь новый
  функционал; 100% покрытие.
- Обновить **README (RU+EN), CLAUDE.md, контекст, спеки** под новое состояние; релиз минор-версии.

## 5. Влияние (сводно)

БД (новые таблицы для AI-гипотез/рисков/планов проверки, история снапшотов) · API (генерация гипотез,
история, settings, refresh) · shared (типы гипотез по новым формулам, риски) · UI (Overview-графики,
фильтры, Settings, History, кнопки) · отчёт (ГОСТ-структура, DOCX/PDF) · конфиг (Settings пишет
локально) · документация и тесты.
